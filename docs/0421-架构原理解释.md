# coworkersfun 架构原理通俗版

> 写给第一次碰 GitHub / Vercel / Supabase 的你，不讲术语，用"餐厅"类比。

---

## 一图流：整个系统

```
[用户浏览器]
     │
     │  打开 https://coworkersfun.vercel.app
     ▼
┌──────────────────────┐
│   Vercel（餐厅前厅+后厨）│
│                      │
│  ① 前厅（静态文件）     │  ← 把你 index.html + js 送到用户浏览器
│                      │
│  ② 后厨（Serverless   │  ← 用户要点赞/登录/编辑时，前厅会叫后厨帮忙
│       Functions）    │     后厨代码就是 api/*.js 里那些文件
└──────────────────────┘
     │
     │  后厨需要查数/改数时，发请求到
     ▼
┌──────────────────────┐
│    Supabase（仓库）     │
│   Postgres 数据库      │ ← 真实存 cards 表、计数、编辑后的文案
└──────────────────────┘
```

## 各角色是啥

### GitHub = 菜谱复印店

你把代码（菜谱）上传到 GitHub，等于"备份 + 授权 Vercel 来拿"。
GitHub 本身不做菜，只存菜谱最新版本。

### Vercel = 你承包的餐厅

做两件事：
1. **发静态文件**：你打开网址，Vercel 把 `index.html` 和打包好的 JS 送给你浏览器 — 这部分是"前厅"
2. **跑 Serverless Function**：你在浏览器点"点赞"按钮时，浏览器偷偷发了个 HTTP 请求到 `/api/vote`，Vercel 启动一个短命的 Node.js 进程跑 `api/vote.js` 这个文件，跑完就关掉。

**为啥叫 "Serverless"？**
因为你不用租一台一直开机的服务器。用户来才开一下，用完就关，按调用次数收钱（免费额度够你玩一辈子）。

### Supabase = 你租的仓库

Postgres 数据库（就是用了几十年成熟的 SQL 数据库），Supabase 帮你托管起来 + 加了个管理界面。
它存的是你那个 `cards` 表里 48 行数据，每行有 id / name / quote / likes_count / dreads_count 等列。

## 一次完整请求的旅程（用"点赞"举例）

```
① 你在手机浏览器点 "这就是我的领导 👍"
    ↓
② 你浏览器里的 card.js 做：
   fetch('/api/vote', { cardId: 'R001', kind: 'like' })
    ↓ HTTPS 请求出发
③ 请求到 Vercel 边缘节点（全球最近一个机房）
    ↓
④ Vercel 看到路径是 /api/xxx，不走前厅，找"后厨"
    ↓
⑤ 启动一个 Node.js 进程，加载 api/vote.js
    ↓
⑥ api/vote.js 读环境变量 SUPABASE_URL + SUPABASE_SERVICE_ROLE_KEY
   用 @supabase/supabase-js 连到 Supabase
    ↓
⑦ 调用 RPC `increment_vote('R001', 'like')`
    ↓
⑧ Supabase 在 cards 表里原子执行 SQL:
   UPDATE cards SET likes_count = likes_count + 1 WHERE id = 'R001'
   返回新的 likes_count 值
    ↓
⑨ Node.js 进程把结果 JSON 返给浏览器 { likes_count: 17 }
    ↓
⑩ 你浏览器看到数字 +1，重绘卡片
    ↓
⑪ Vercel 关掉这个 Node 进程（Serverless 生命结束）
```

第 ⑪ 步是重点 — 这个 Node 进程只活了几百毫秒。下一次有人点赞，从 ⑤ 重新来一遍。

## 为啥不让浏览器直接连 Supabase（必须走 Vercel 后厨）

你可能会问："浏览器也能直接发 HTTP 请求，为啥还要绕一圈 Vercel？"

**原因 1：密钥不能暴露在前端**

- `SUPABASE_SERVICE_ROLE_KEY` 这把钥匙能做**任何**事（删表、改任何行、读所有数据）
- 如果放在前端 JS 代码里，任何用户 F12 打开控制台就能偷到
- 放到 Vercel 后厨里 → 只有 Vercel 的服务器环境看得到，浏览器看不见

**原因 2：权限逻辑**

- 编辑卡牌要管理员密码对才行 —— 这个"对比密码"的逻辑必须在**你信任的服务器**跑，不能在浏览器里跑（浏览器里的代码用户可以随便改）

所以架构是：
```
浏览器（不信任，任何人能看代码）
    ↕ 只能调用 /api/*（你定义好的几个受控端点）
Vercel 后厨（信任，只跑你写的代码）
    ↕ 拿 service_role key（万能钥匙）
Supabase（信任，你的数据仓库）
```

## 环境变量为啥要填那 4 个

每个变量解决一个问题：

| 变量 | 为啥要 |
|---|---|
| `SUPABASE_URL` | 告诉后厨"仓库在哪"（Supabase 项目的域名） |
| `SUPABASE_SERVICE_ROLE_KEY` | 万能钥匙，后厨拿着才能进仓库操作（前端永远不会见到它） |
| `ADMIN_PASSWORD` | 管理员登录密码。改代码就能改的话太死板；env var 能在 Vercel 随时改 |
| `ADMIN_SECRET` | 给管理员 cookie **签名**用的盐。没有这个，任何人都能伪造一张"我已登录"的饼干 |

**为啥不写死在代码里？**
因为代码上传到 GitHub 是公开的，密码和密钥进代码 = 满世界公开。
env var 只存在 Vercel 项目设置里，不进 git。

## 管理员登录的"cookie 签名"是什么黑魔法

你输密码对了之后，服务端给你浏览器发了一个 cookie：
```
admin_session=MTc3Njc5MjAzMDM1NQ==.f7ce451d3638dcc7...
```

拆开看：
```
   base64(过期时间戳)  .   HMAC-SHA256(时间戳, ADMIN_SECRET)
```

下次你访问 `/api/admin/card` 时浏览器自动带这个 cookie，服务端：
1. 拆出时间戳，看有没有过期
2. 拿 `ADMIN_SECRET` 重新算一遍 HMAC，比对一致 → 是我签发的，放行

**为啥这样安全？**
HMAC 需要 `ADMIN_SECRET` 才能生成。攻击者不知道 secret，改时间戳/伪造 cookie 都会算出不一样的 HMAC，服务端一验就发现。

**为啥 2 小时过期？**
短 TTL 让偷到 cookie 的攻击者窗口有限；2 小时够管理员一次性把卡牌改完。

## 为啥要 `data/cards.json` 和 Supabase 两份数据

这个设计是故意的：

- `data/cards.json` 是**初始种子**，写在代码仓库里
- Supabase 是**运行时真数据**（带 likes_count / dreads_count，还能编辑）
- `scripts/gen-seed.js` 把 JSON 转成 `db/seed.sql`（一堆 UPSERT 语句）
- 部署流程：在 Supabase 跑一次 seed.sql → 48 行数据落地

好处：
1. 卡牌原始清单被 git 追踪（版本化）
2. 线上运行时可以改（通过 admin 面板改 quote），不影响 git 历史
3. 将来清空重来，再跑一次 seed.sql 就回到初始状态

## 还有什么坑你今后可能踩

**1. 改代码后要重新部署**
改 `src/` 或 `api/` 里的文件，`git push` 到 GitHub → Vercel 自动检测到新提交 → 自动重新部署。
但改**环境变量**不自动触发部署，需要手动 Redeploy。

**2. Vercel 部署失败日志在哪看**
Dashboard → 你的项目 → Deployments → 点那条红色失败的 → Logs。
Serverless Function 运行时报错（比如 Supabase 连不上）在 **Runtime Logs** 里。

**3. Supabase 数据要改 schema 怎么办**
直接在 SQL Editor 写 `ALTER TABLE ...` 即可。
但建议：改 `db/schema.sql` 文件 → 同步跑到 SQL Editor。这样文件和线上保持一致。

**4. 怎么看谁点了多少赞**
目前没做 votes 事件表（spec 里的 YAGNI 决策），所以只能看聚合总数，看不了谁点的。
要加的话加一张 votes 表，每次点赞 `insert into votes (card_id, ip, ts)`，然后再写个 admin 端点查询。

## 一句话总结

**你的代码躺在 GitHub；Vercel 从那里抓来，前端发给用户、后端跑 serverless function；Supabase 是真数据库，只让 serverless function 用密钥访问。**

三个服务各司其职，你负责写代码 + 配置 4 个环境变量。
